IMPORTANT RULES
===============

1. Do not use one giant Windows batch command.
2. Run each command one at a time.
3. Windows CMD commands are run from Windows Command Prompt.
4. Linux commands are run after SSH login to the server.
5. Do not run the rsync command until the verification commands show the app files.
6. DEVSERVER.odt is not part of deployment.
7. The .env file on the server is preserved and should not be overwritten by upload.
8. PDFs/documents and generated contracts are still file-based. Staff users, customer accounts, business settings, imported lookup data, contracts, estimates, and drafts use PostgreSQL when DATABASE_URL is set.
9. Do not expose PostgreSQL, MySQL, or the Node app port directly to the internet.
10. A fresh install with no staff users now opens /setup for first-run business/admin setup.

PART 6 - LINUX SERVER: VERIFY STAGING BEFORE COPYING
====================================================

Run these one at a time:

PART 14 - WHAT SHOULD BE OPEN
=============================

Normal production access should be:

Open:

Port 80 - Nginx HTTP redirect/proxy

Port 443 - Nginx HTTPS

Not directly open from Windows or the internet:

Port 3000 - Node app, private behind Nginx

Port 3306 - MySQL, private/local only if installed

Port 5432 - PostgreSQL, private/local only if installed

If the direct app health check works on the server:

curl http://127.0.0.1:3000/api/health

that means the app is running.

If this Nginx test returns a 301 redirect:

curl -H "Host: contracts.edgefam.com" http://127.0.0.1/api/health

that usually means Nginx is redirecting HTTP to HTTPS. To test HTTPS locally on the server, run:

curl -k --resolve contracts.edgefam.com:443:127.0.0.1 https://contracts.edgefam.com/api/health

You want to see:

{"ok":true}


PART 15 - POSTGRESQL LOOKUP SETUP
=================================

The portal now supports PostgreSQL for live record data:

customers
suppliers
products
import_runs
contract_packets
estimate_records
contract_drafts
staff_users
customer_accounts
portal_settings

Generated PDFs, uploaded PDFs, OCR staged documents, and packet files are still stored as server files.

When DATABASE_URL is set, staff users, customer portal accounts, and business settings are shared through PostgreSQL. The JSON files under data/settings are backup/migration files only.

When DATABASE_URL is set, the app creates the PostgreSQL tables on startup.

To make the Preimport tab write shared lookup data to PostgreSQL, edit the server .env:

sudo nano /opt/apps/customerportal/app/.env

Add or confirm this line:

DATABASE_URL=postgresql://customer_portal_user:YOUR_POSTGRES_PASSWORD@localhost:5432/customer_portal

Then restart:

sudo systemctl restart customerportal

For a true first store-user setup on a fresh install, open:

https://contracts.edgefam.com/setup

If staff users already exist, /setup redirects to the staff login page.

First-run setup creates the first manager/admin username and password.

After that first admin logs in, additional staff users are created at:

Admin Menu > Users

Use this for Jamie/Billy/Michelle or any future store employee:

Name
Username
Temporary password
Manager/admin checkbox if they can manage users/settings
Must change password checkbox
Disabled checkbox for accounts that should no longer log in

Fresh installs do not automatically create Edgewater demo users unless PORTAL_SEED_STAFF=true is set in .env.

After login, go to:

Admin Menu > Settings > Preimport

Admin > Settings > Preimport should say:

Storage: PostgreSQL shared lookup tables

If DATABASE_URL is blank or missing, the Preimport tab falls back to local files under:

data/preimport


PART 16 - OCR SETUP CHECK
=========================

Admin Menu > Settings > Preimport can stage PDFs/images and run OCR.

The server should have these available:

ocrmypdf
tesseract
python with pypdf

Optional .env values:

OCRMYPDF_BIN=ocrmypdf
TESSERACT_BIN=tesseract
PDFTOTEXT_BIN=pdftotext
OCR_LANGUAGE=eng
OCR_TIMEOUT_MS=180000

If OCR fails on a scanned PDF, check logs:

sudo journalctl -u customerportal -n 120 --no-pager


PART 17 - POSTGRESQL UI FOR SOMEONE USED TO MYSQL
=================================================

Recommended Windows database UI:

DBeaver Community

Why:

It feels familiar if you are used to MySQL tools, works with PostgreSQL and MySQL, and supports SSH tunnels.

Do not connect to PostgreSQL by opening port 5432 publicly.

Use an SSH tunnel instead.

Option A - DBeaver built-in SSH tunnel:

Main PostgreSQL settings:

Host: 127.0.0.1
Port: 5432
Database: customer_portal
User: customer_portal_user
Password: the PostgreSQL password from the server .env or setup notes

SSH tunnel settings:

SSH host: 192.168.1.70
SSH port: 22
SSH user: michelle-work
Remote database host: 127.0.0.1
Remote database port: 5432

Option B - Manual SSH tunnel from Windows, then DBeaver:

Run this from Windows Command Prompt or PowerShell:

ssh -L 15432:127.0.0.1:5432 michelle-work@192.168.1.70

Leave that SSH window open.

Then connect DBeaver to:

Host: 127.0.0.1
Port: 15432
Database: customer_portal
User: customer_portal_user

Backup UI:

pgAdmin is the official PostgreSQL admin tool, but DBeaver is usually easier when moving from MySQL because it can manage both database types in one place.


PART 18 - IS THE DATABASE EMPTY ON A FRESH INSTALL?
==================================================

Yes. A fresh install starts with no customer contracts, no generated PDFs, no saved business settings, and empty PostgreSQL lookup tables unless you import data.

The app currently stores live portal data in these folders:

data/packets
data/generated
data/settings
data/logs
data/estimates
data/estimate-module
data/preimport

The upload archive intentionally excludes those folders so deployment does not overwrite real server data.

The deploy rsync command also excludes those folders for the same reason.

On a brand-new install, the app creates the folders when needed, but they start empty.

PostgreSQL is currently used for live record data when DATABASE_URL is set:

customers
suppliers
products
import_runs
contract_packets
estimate_records

Generated PDFs, settings, OCR staged documents, and watched estimate PDF files are still file-based on the server.

COMMON PROBLEMS
===============

Problem:
scp says it cannot resolve hostname F.

Cause:
You ran scp from inside the Linux SSH session, or scp got confused by a Windows F: path.

Fix:
Exit SSH, return to Windows CMD, run:

cd /d F:\

Then run:

scp customerportal-upload.tgz michelle-work@192.168.1.70:/home/michelle-work/uploads/customerportal-upload.tgz


Problem:
sudo says a terminal is required.

Cause:
You tried to run the server rsync/restart commands through a remote quoted ssh command.

Fix:
SSH into the server first:

ssh michelle-work@192.168.1.70

Then run the Linux commands directly on the server.


Problem:
npm ci says package-lock.json is missing.

Cause:
The staging folder is empty or incomplete, or the wrong folder was copied.

Fix:
Stop. Do not rsync. Re-run the upload and verify these files before rsync:

ls ~/uploads/customerportal/package.json
ls ~/uploads/customerportal/package-lock.json
ls ~/uploads/customerportal/server/index.js
ls ~/uploads/customerportal/public/home.html


Problem:
CSS or images are missing through the website, but work on port 3000.

Cause:
Nginx location block may be wrong. Earlier issue was using location = / instead of location /.

Fix:
Check the Nginx site file and make sure the main app proxy uses:

location / {

not:

location = / {
